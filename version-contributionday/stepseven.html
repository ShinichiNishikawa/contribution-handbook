Theme Review

Required items

The following must be met by all themes before they can be approved to be included in the WordPress.org theme repository. Along with these checks you should also make sure you run the theme through the theme check plugin. It automatically checks for all the required items.

Accessibility

  • If the theme has the tag 'accessibility-ready' then it needs to meet these requirements.

Code

  • Must not generate any PHP or JS errors.
  • Must include as a minimum index.php and style.css.
  • Should have a valid DOCTYPE declaration and include language_attributes.
  • Everything must be sanitised.
  • No removing or modifying non-presentational hooks.
  • No shortcodes are allowed.
  • Support the following WordPress-generated CSS classes:
    • alignleft
    • alignright
    • wp-caption
    • wp-caption-text
    • gallery-caption
    • sticky (can be unstyled)
    • bypostauthor (can be unstyled)

Core Functionality and Features

  • Use WordPress functionality and features first, if available.
  • Shouldn't include admin/feature pointers.
  • No custom post types and no custom taxonomies.
  • No pay wall restricting any WordPress feature.
  • No disabling of the admin tool bar.
  • Use get_template_directory() rather than TEMPLATEPATH to return the template path.
  • Use get_stylesheet_directory() rather than STYLESHEETPATH to return the stylesheet path.
  • Use function parameters, filters, and action hooks where appropriate in order to modify content, rather than hard-coding. For example wp_title should be modified using a filter.
  • Able to have child themes made from them.
  • Comments_template() is required.

Documentation

  • Provide sufficient documentation to explain the use of any custom features or options or any limitations (for example menu restrictions).

Favicons

  • If implemented, favicons must be disabled by default and have the ability for users to over-ride.

Language

  • All front-facing strings should be translatable.
  • Must use a single unique theme slug – as the theme slug appears in style.css. If it uses a framework then no more than 2 unique slugs.
  • Can use any language for front facing text. However, they should only use the same one for all text.

Licensing

  • Be 100% GPL and/or 100% GPL-compatible licensed.
  • Declare their copyright and license explicitly. They should also use the license and license uri header slugs to style.css.
  • Any resources included such as fonts or images should have all license information declared.
  • All code and design should be your own or legally yours. Cloning of designs is not acceptable.

Naming

  • Theme names must not use: WordPress, Theme.
  • Must spell “WordPress” correctly in all public facing text: all one word, with both an uppercase W and P.

Options and Settings

  • Save options in a single array.
  • Use sane defaults and must not write default setting values to the database.
  • Use edit_theme_options capability for add_theme_page(), rather than rely on a role (e.g. “administrator”), or a different capability (e.g. “edit_themes”, “manage_options”) for the capability to add the settings page.
  • If using a theme options page, it should be under appearance.
  • Use either the Settings API or the Customizer for implementing theme options.
  • Prefix all options, custom functions, custom variables, and custom constants with theme-slug (or appropriate variant).

Plugins

  • Shouldn't include any plugins. A theme can recommend plugins but the plugins should not be included in the theme code.
  • Themes shouldn't do things considered plugin territory.

Screenshot

  • Screenshot.png should be of the actual theme as it appears with default options, not a logo or mockup.
  • The screenshot.png should be no bigger than 880x 660px.

Security and Privacy

  • Must not phone home without informed user consent. Find out more about security here.
  • Any collection of user data should be ‘opt-in’ only and have a theme option that is set to disabled by default.
  • Validate and sanitize all untrusted data before entering data into the database, and to escape all untrusted data before being output in the Settings form fields or in the Theme template files (see: Data Validation)
  • Use esc_attr() for text inputs and esc_html() for textareas.

Selling, credits and links

  • If you are a themeshop you should be selling under GPL to be in the WordPress.org repo.
  • If the theme adds a footer credit link, there should only be one.

Stylesheets and Scripts

  • No hard coding of scripts, styles and Favicons unless a browser workaround script.
  • No analytics or tracking.
  • No minification of scripts or files unless provide original files.
  • Required to use core-bundled scripts rather than including their own version of that script. For example jQuery.
  • Include all scripts and resources it uses rather than hot-linking. The exception to this is Google libraries.
  • If a tag is used in style.css the theme should support that feature or adhere to what that tag stands for. For example custom background or header.

Templates

  • If using templates, custom template files should be called using get_template_part() or locate_template().
  • There are several template specific things you should consider when certain ones are being used.
  • You should use *_url() template tags, rather than bloginfo() equivalents.
It's worth noting we are working to automate a lot of the above requirements.
